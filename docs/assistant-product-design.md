# 小白助手 产品设计文档

## 文档目标

定义“小白助手”的产品定位、用户场景、能力边界、交互方式与阶段目标。

本文件只讨论产品设计，不讨论具体代码结构、依赖、构建方式与文件落点。

## 当前状态

- 状态: 已实现 V1 原型
- 当前实现阶段: V1 前端版
- 本阶段目标: 做出一个可在酒馆内直接使用的技术支持助手
- 当前体验进度:
  - 已支持会话持久化
  - 已支持工具调用中止
  - 已支持工具过程状态提示
  - 已支持清空对话与保存反馈
  - 已支持基础使用引导与示例提问

## 产品定位

小白助手是挂在小白 X 内部的“技术支持型 Agent”。

它的主要职责不是陪聊，不是读剧情，不是扮演角色，而是帮助用户在酒馆里自助解决与下列内容相关的问题：

- 小白 X 插件本身的功能、配置、报错与行为解释
- SillyTavern 前端侧的代码、扩展、设置与交互机制
- 用户自己在使用过程中碰到的“为什么会这样”和“应该怎么配”

一句话定义:

小白助手 = 内置在小白 X 里的、能查代码和文档的技术支持助手。

## 核心原则

### 1. 少绕路

能直接回答就直接回答，能直接查代码就直接查代码，不做多余中间层，不把简单事情搞成工作流表演。

### 2. 不手搓协议

有官方 SDK 的模型接入一律优先用官方 SDK。

### 3. 读尽量宽，写尽量收

第一版允许助手尽可能完整地理解前端可见代码与公开资源；写入能力严格收敛，不允许修改代码。

### 4. 先做真正有用的版本

先解决“用户能不能自己查问题、问问题、拿到靠谱回答”，而不是先堆大量花哨能力。

### 5. 心智统一

用户应当清楚知道：

- 小白助手是查问题的
- 它会用工具
- 它默认只读代码与资料
- 它不会偷偷改插件代码

## 目标用户

- 普通用户: 不懂代码，但想知道某个报错、选项、行为是什么意思
- 轻度折腾用户: 能看路径、看设置，但不想来回翻代码
- 维护者自己: 需要快速定位用户问题、复查行为、验证某个功能入口

## 主要使用场景

### 场景 1: 解答问题

示例:

- “为什么我的 L0 一直 429？”
- “这个设置项是干什么的？”
- “为什么勾选后重启又失效？”
- “OpenRouter 这边拉不到模型是什么原因？”

期望行为:

- 助手优先查代码或配置再回答
- 回答尽量具体，带路径、模块名、设置项名
- 不装懂，不胡猜

### 场景 2: 帮用户自查功能

示例:

- “小白 X 的 TTS 设置是从哪走的？”
- “剧情总结的 L0 请求现在是不是直连？”
- “这个按钮点了之后会触发什么？”

期望行为:

- 助手可列出相关文件
- 可读文件并引用关键代码位置
- 可解释调用链

### 场景 3: 生成工作记录

示例:

- “把这次排查结果整理一下”
- “给我写一份测试记录”

期望行为:

- 助手可把结果写入一个受控工作区
- 写出的内容是文档，不是代码修改

## 非目标

以下不是 V1 的目标:

- 聊天陪伴型 AI
- 角色扮演助手
- 自动修改酒馆或插件代码
- 自动改世界书、聊天记录、角色卡
- 替代剧情相关模块

## V1 功能范围

### 必做

- 在小白 X 内提供一个独立入口打开“小白助手”
- 助手有独立聊天界面
- 支持至少三类模型接入:
  - OpenAI-compatible
  - Anthropic
  - Google AI
- 支持工具调用
- 支持只读查阅:
  - 小白 X 扩展源码
  - SillyTavern 前端公开源码与资源路径
- 支持受控写入工作区文档
- 回答时保留技术支持导向

### V1 不做

- 修改任何现有代码文件
- 修改聊天内容或世界书内容
- 读 SillyTavern 服务端私有源码目录
- 自动执行高风险动作

## 可读范围

V1 明确支持:

- 小白 X 扩展目录中的源码与文本文件
- SillyTavern `public` 目录下可由前端直接访问的源码与资源路径
- 助手自己的工作记录文件

V1 明确不承诺:

- `src/` 等服务端私有目录的直接读取
- 任意本地磁盘路径读取

说明:

这不是主观限缩，而是当前第三方前端扩展的可分发边界。若后续采用干净的服务端插件方案，再扩展深度读取能力。

## 可写范围

V1 仅允许写入助手工作区文档。

允许:

- 新建或覆盖助手工作记录
- 导出排查结果、摘要、笔记

不允许:

- 写入源码
- 覆盖插件设置页代码
- 修改用户聊天数据
- 修改世界书、角色卡、主程序文件

## 入口设计

V1 提供两个入口:

- 小白 X 设置页中的“启用小白助手”开关
- 小白 X 设置页中的“打开助手”按钮

不额外引入复杂浮层入口，不先铺一堆悬浮按钮。

## 交互设计

### 基础交互

- 用户打开助手
- 助手显示历史会话
- 用户输入问题
- 助手自行决定是否调用工具
- 回答中尽量引用文件路径、模块名、设置项名
- 运行中可直接点“终止”中断当前轮
- 清空对话后不影响配置
- 空白会话时显示示例问题，帮助用户快速上手

### 配置交互

助手面板内自行管理:

- 模型提供商
- Base URL
- Model
- API Key
- Workspace 文件名

不把这些细碎配置塞进主设置页，避免主设置页膨胀。

### 工具交互

用户不需要手动点“调用哪个工具”。

助手自动决定何时:

- 列文件
- 搜文件
- 读文件
- 写工作记录

当前 UI 额外要求:

- 让用户看到当前正在做什么，例如“正在搜索关键词 …”“正在读取文件 …”
- 工具结果默认用人能看懂的摘要显示
- 详细原始结果可以折叠展开，不把聊天区直接刷成大段 JSON

## 回复风格要求

- 面向普通用户时说人话
- 面向排查时具体、可核对
- 不要虚张声势
- 不确定时明确说明“我需要再查”
- 不要把技术支持回答写成论文

## 成功标准

V1 完成后，下面这些事情应能直接完成:

- 用户在插件里打开助手并正常聊天
- 用户配置三类模型中的任意一种后，助手可正常回答
- 助手可列出相关文件并读取内容
- 助手可根据代码内容回答问题
- 助手可写出排查记录到工作区文档
- 用户关闭助手再打开，不会直接丢失整段会话
- 用户能明确知道当前是否在请求模型、调用工具或已终止

## 后续阶段

### V2

- 引入更强的代码检索能力
- 增加更稳定的工作区索引
- 增加更明确的错误诊断卡片

### V3

- 若分发与安装链路干净，再引入服务端只读桥
- 在不破坏分发体验的前提下扩展对 `src/` 的读取
